== Set a Display Name for a Parameter

To select a display name for a parameter, you must use the
`displayName` property under the parameter node. It applies for any
parameter type. You can define parameters using the `queryParameter`,
`uriParameter`, or `header` nodes, and they refer to the corresponding
parameter defined in the API spec.

[source,yaml]
----
endpoints:
   /content_search:
      operations:
         get:
            pagination: content-search-pagination
            displayName: Get Content Search
            description: Provides keyword and refinement search functionality for content assets.
            expects:
               queryParameter:
                  count:
                     displayName: count
                     description: The maximum number of instances per request.
----

== Set a Description for a Parameter

To select a description for a parameter, you must use the `description`
property under the parameter node. It applies to any parameter type.
Parameters can be defined under the `queryParameter`, `uriParameter`, or
`header` nodes, and they refer to the corresponding parameter defined in
the API spec.

[source,yaml]
----
endpoints:
   /content_search:
      operations:
         get:
            pagination: content-search-pagination
            displayName: Get Content Search
            description: Provides keyword and refinement search functionality for content assets.
            expects:
               queryParameter:
                  count:
                     displayName: count
                     description: The maximum number of instances per request. Default value is 25.
----

The parameter description then appears as a tooltip beside the parameter in Anypoint Studio:

== Ignore a Parameter

Parameters can be ignored when defined in both operations and security
schemes. +
When a parameter is ignored in the connector descriptor, it is not
present in the generated connector.

To ignore a parameter, you must use the `ignored` property under the
parameter node. It applies for any parameter type.

[source,yaml]
----
security:
 oauth2:
   headers:
     Authorization:
       ignored: true

endpoints:
   /content_search:
      operations:
         get:
            expects:
               queryParameter:
                  count:
                     ignored: true
----

In the previous example, the `Authorization` header was ignored from the
`oauth2` security definition as it is automatically generated by REST SDK
when there is an OAuth scheme defined. The example also shows how to
ignore a query parameter from an operation.

== Select a Query Parameter Array Format

Query parameter arrays are handled in different ways in different REST
API implementations.

Currently, REST SDK supports 2 formats: `comma` and `multimap`.

REST SDK uses the multimap format by default if you donâ€™t add any
information to the connector descriptor, and in it every value of a
query parameter that is an array is sent by repeating the query
parameter key for each value in the query string, such as
`?key=value1&key=value2&key=value3`.

The comma format sends the array values using the same key and split by
commas, such as `?key=value1,value2,value3`.

You can select the query parameter array format globally for the
connector or at an operation level. In both cases, the query parameter
is defined by using the `queryParameterArrayFormat` property.

[source,yaml]
----
queryParameterArrayFormat: multimap

endpoints:
 /operation/{names}:
   operations:
     get:
       queryParameterArrayFormat: comma
----

In this example, the `GET /operation/\{names}` operation must use comma
query param array format, but that all the other operations defined in
the connector must use the multimap format.
